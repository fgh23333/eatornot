## Slide 4：系统架构全景（40秒）⭐ 关键页

**【画面】** 2×2 网格架构图：Orchestrator → 并行分析 → 辩论 → 输出

**【说】**

> 来看我们的系统架构。整个决策流程分为四个阶段：
>
> **第一，Orchestrator 智能调度。** 我们利用 Gemma 4 的 Function Calling 能力，让大模型自己分析用户输入，从 8 个候选 Agent 中动态唤醒 3 到 5 个最相关的。不是硬编码规则，是 LLM 智能调度。
>
> **第二，Agent 并行分析。** 被选中的 Agent 通过 asyncio.gather 并发执行，减脂、营养、预算、食欲……每个 Agent 从自己的专业角度给出独立分析。
>
> **第三，圆桌辩论。** 这是我们的核心创新。各 Agent 的意见可能矛盾 — 减脂Agent说别吃，食欲Agent说该犒劳。DebateEngine 通过 4 个阶段让它们"吵"出最优解。
>
> **第四，三方案输出。** 最终给出自律、省钱、犒劳三套差异化方案，附带每个 Agent 的投票结果，用户自己做最终选择。

**（加重语气）** "注意，每次决策大约需要 **10 到 12 次 LLM 调用**，不是单次 Prompt。"

## Slide 5：Function Calling（40秒）⭐ 关键页

**【画面】** 左右分栏：左边是说明文字 + Agent 标签，右边是代码日志

**【说】**

> 这页展示我们对 Gemma 4 Function Calling 的深度利用。
>
> **左边**：传统做法是 IF-ELSE 关键词匹配，"减肥"就调减脂Agent。但用户说"我很累想吃好的，但在减肥"，这种复杂情绪关键词抓不住。我们让大模型通过原生 Tool Calling 自主选择 Agent 组合。
>
> **右边是真实的执行日志**（指向代码块）：
> - Orchestrator 先看到 8 个候选 Agent
> - 然后通过 Function Calling 动态选择了 weight_loss、budget、craving 三个
> - 下方的 MCP 闭环：query-nearby-stores 查门店 → query-meals 查菜单 → calculate-price 算价格 → create-order 创建订单
>
> **全链路都是 Gemma 4 自主选择工具，不是硬编码。而且这些 MCP 工具连的是麦当劳真实接口** — 比如搜"上海"会返回南京东路悦荟广场、福州路书城这些真实门店，菜单里的巨无霸是真实在售的 25 块 5。不是 Mock 数据。

## Slide 7：圆桌辩论引擎（30秒）⭐ 关键页

**【画面】** 时间线：R1→R2→R3→R4

**【说】**

> 这是整个系统最核心的 DebateEngine。四个阶段：
>
> **R1 初始立场**：各 Agent 独立陈述。比如减脂Agent说热量超标必须吃沙拉，预算Agent说套餐超支。
>
> **R2 发现冲突**：Gemma 4 分析矛盾 — 减脂说热量超限，但食欲Agent指出用户极度疲劳需要碳水满足感。这俩打起来了。
>
> **R3 形成妥协**：主持人综合意见，找到帕累托最优 — "高蛋白单品组合加无糖可乐，22 块搞定"。
>
> **R4 最终投票**：每个 Agent 对方案投票，approve、warn 或 reject。最终产出三套方案。
>
> **流程框架是工程固定的，但每个阶段的分析内容由 Gemma 4 动态生成。**

## 🎤 QA 预案（3分钟）

### 高概率问题（必准备）

**Q1: 为什么选 Gemma 4 31B 而不是更小的模型？**
> 31B Dense 在三个维度表现最好：中文营养学知识理解、多步推理一致性、Function Calling JSON 输出稳定性。我们尝试过 12B，在辩论环节的多 Agent 上下文保持上不够稳定，容易出现角色混淆。

**Q2: Orchestrator 和传统 rule-based 路由有什么区别？**
> 传统方式用关键词匹配，"减肥"→减脂Agent。但用户说"我很累想吃好的但在减肥"，关键词无法捕获"累"→食欲Agent + "减肥"→减脂Agent 的组合需求。我们的 Orchestrator 用 Gemma 4 分析完整上下文，能处理这种多意图的复杂场景。

**Q3: 辩论引擎的 4 阶段是硬编码还是 LLM 驱动？**
> 混合模式。流程框架（4个阶段）是工程固定，保证流程可控。但每个阶段的内容 — 发现哪些冲突、如何妥协、投什么票 — 完全由 Gemma 4 动态生成。这样既保证稳定性，又保持灵活性。

**Q4: 和通用 AI 助手（ChatGPT/Gemini）的饮食建议有什么区别？**
> 三个核心差异：第一，我们是多 Agent 并行分析，通用助手是单次对话。第二，我们有 4 阶段辩论引擎，能发现多维度冲突并形成妥协方案。第三，我们有跨会话的记忆学习 + MCP 下单闭环，通用助手只给建议不能执行。

**Q5: 怎么衡量推荐质量？**
> 目前通过两个指标：用户反馈评分（1-5分）和方案采纳率。长期计划引入营养师标注数据做基准测试，量化推荐的营养合理性和用户满意度。

**Q5.5: 现场能演示吗？数据是真的吗？**
> 全是真的。前后端都已部署上线 — 前端在 Cloudflare Pages，后端在 Cloudflare Workers。我现在打开页面，搜索"上海"，会实时调麦当劳 MCP 返回南京东路、福州路这些真实门店；点开任一门店能看到 159 个真实菜品，巨无霸 25 块 5、板烧鸡腿堡 23 块 5 都是当前在售价。不是录屏，不是 Mock，是 live data。

### 中等概率问题

**Q6: MCP 接入有什么挑战？**
> 麦当劳 MCP 用的是 Streamable HTTP + JSON-RPC 协议，而且它返回的 tool result 是"说明文字 + 嵌套 JSON 字符串"的混合体，不能直接 JSON.parse。我们在 Worker 里写了专门的解析器，能从混合文本里提取出真正的门店和菜单数据。另外做了 Mock 降级机制 — MCP Token 不可用时自动切回模拟数据，确保演示在任何情况下都不挂。现在线上版搜"上海"能返回 5 家真实门店、159 个真实菜品和真实价格。

**Q7: Agent Memory 怎么避免冷启动？**
> 新用户先用快速建档模式，只需选目标、预算、饥饿感，首次决策不依赖历史记忆。随着用餐反馈积累，大约 3-5 次后，记忆系统开始影响 Agent 权重和推荐偏好。

**Q8: 系统延迟怎么样？每次 10-12 次 LLM 调用需要多久？**
> Agent 通过 asyncio.gather 并行执行，6-8 个 Agent 同时跑。实测一次完整决策大约 8-15 秒，其中 LLM 调用是主要耗时。辩论引擎是串行的 4 个阶段，但每个阶段只有一次 LLM 调用。整体在可接受范围内。

**Q9: 安全性怎么保证？过敏、极端饮食？**
> Safety Agent 有最高优先级 — 即使 Orchestrator 没选它，只要检测到过敏相关关键词就会强制激活。另外系统永远不会自动下单，所有订单都需要用户明确确认。对于极端饮食行为（如连续多天低于基础代谢），系统会发出警告。

**Q9.5: 线上系统怎么监控和排查问题？（可观测性）**
> 全链路可观测。后端跑在 Cloudflare Workers 上，每次请求都打结构化日志 —— Gemini 调用的状态码和耗时、MCP 请求是否成功、哪个 Agent 被激活、辩论各阶段的结果，全部能通过 Workers Observability 实时查看和按字段筛选。AI Gateway 这一层还能看 Gemma 4 的 token 用量、成功率、p99 延迟。所以"推荐慢了"、"MCP 报错"这类问题能在日志里直接定位到具体哪一步、哪一次调用，不是黑盒。这对一个有 10+ 次 LLM 调用、多 Agent 并行的系统来说，没有可观测性根本没法 debug。

**Q10: 为什么选择麦当劳作为第一个接入平台？**
> 两个原因：第一，麦当劳有成熟的 MCP 接口和完整的营养数据，方便我们快速实现闭环。第二，程序员群体高频消费麦当劳，是最真实的场景。而且我们已经**真的跑通了线上闭环** — Worker 直连麦当劳 MCP，门店搜索、菜单查询、价格计算全是真实数据。扩展到美团、饿了么只需要实现 FoodProvider 抽象接口。

**Q13: 数据隐私怎么处理？**
> 用户数据全部存储在本地 SQLite，不发送到第三方。LLM 调用只传递脱敏后的分析请求，不包含用户身份信息。符合最小数据原则。

**Q14: 如果 Gemma 4 API 挂了怎么办？**
> 三层降级：LLM 调用失败 → Orchestrator 降级到关键词匹配 → Agent 分析降级到规则引擎。系统在任何情况下都能给出基本建议，不会完全崩溃。

**Q15: 为什么不用 LangChain 或其他 Agent 框架？**
> 我们的核心是 4 阶段辩论引擎，需要精细控制 Agent 调度、权重调节和算法降级策略。现有框架（LangChain、CrewAI 等）的抽象层级更适合单轮工具调用型 Agent，不太适合我们的多轮结构化辩论场景。所以选择了基于 Gemma 4 Function Calling 的自定义编排。同时保留了 ADK 兼容层，方便未来对接 Google 生态的工具链。
