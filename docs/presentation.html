程序员的"吃"难题

🕐 忘记吃饭

高强度编码进入 Flow 状态，彻底模糊时间概念，经常错过正常饭点。

🍔 情绪性进食

面对 Bug 压力产生补偿性进食欲望，高热量摄入破坏健康目标。

💰 预算失控

随意点餐缺乏统筹，隐蔽且频繁的开销导致月度餐费严重超标。

→ 现有工具只给菜单列表，不理解你的健康目标、预算限制和情绪状态。

核心定位：多 Agent 饮食辩论

传统工具是单维度推荐排序算法。EatOrNot 部署 8 个专业 Agent 从不同角度辩论，给出有理有据的饮食建议。

对比维度 传统外卖平台 EatOrNot 智能体
决策大脑 单一推荐排序算法 8 个专业 Agent 并行分析
评估标准 仅看价格、距离、销量 健康 + 预算 + 情绪 + 安全
个性化 千人一面的通用推荐 学习你的饮食习惯，越用越懂你
业务闭环 推荐完即结束，需手动操作 分析 → 辩论 → 自主下单闭环

多 Agent 协作架构

① Orchestrator 智能调度

基于 Gemma 4 Function Calling 分析用户输入，从 8 个候选 Agent 中动态唤醒 3-5 个最相关的主体。LLM 失败时自动降级到关键词匹配。

② Agent 并行分析

减脂、营养、预算、食欲、时间、安全等 Agent 通过 asyncio.gather() 并发执行，极速收集多维数据，大幅降低决策延迟。

③ Debate 圆桌辩论

经历 R1 初始意见 → R2 发现冲突 → R3 形成妥协 → R4 最终投票，从矛盾需求中寻找帕累托最优解。

④ 三方案输出 + 下单闭环

产出 💪自律 / 💰省钱 / 🍔犒劳 三套差异化方案。用户选定后通过麦当劳 MCP 一键创建订单，全程无需离开系统。

每次决策 ≈ 10-12 次 LLM 调用，不是单次 Prompt 工程

Gemma 4: Native Function Calling

拒绝硬编码，拥抱动态路由

Orchestrator 不用 IF-ELSE 关键词匹配，而是利用 Gemma 4 原生工具调用能力，让大模型自主分析复杂情绪场景并选择 Agent 组合。

例如："很累想吃好的，但在减肥" → 同时激活 减脂 + 预算 + 食欲三个 Agent。

4 阶段圆桌辩论引擎

R1: 初始立场

各 Agent 基于数据与知识库陈述独立分析意见。如：减脂Agent 认为热量超标，预算Agent 发现超支。

R2: 发现冲突

🔥减脂："热量超限！"
🍫食欲："但用户极度疲劳，需要碳水满足感。"
Gemma 4 识别这些矛盾。

R3: 形成妥协

主持人综合各方意见寻找帕累托最优解："高蛋白单品组合 + 无糖可乐，¥22 内搞定。"

R4: 最终投票

各 Agent 对方案投票 (approve/warn/reject)，产出三套差异化方案供用户选择。

技术栈 & 三大核心创新

🧠 LLM 驱动编排器

摒弃 Rule-based 规则引擎，利用 Gemma 4 Function Calling 智能调度专业 Agent，每次决策 10-12 次 LLM 调用。

💬 多 Agent 辩论

不只是简单投票。4 阶段辩论：发现冲突 → 妥协 → 投票，从矛盾需求中计算帕累托最优解。

🔧 Tool Calling 闭环

从营养查询到价格计算再到自主下单，全链路 MCP 接入。麦当劳 MCP 真实数据 + Mock 降级保障。
